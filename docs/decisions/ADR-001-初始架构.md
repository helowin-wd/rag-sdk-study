# ADR-001: 初始项目架构

**状态：** 已采纳

**日期：** 2026-06-01

## 背景

我们正在从头构建一个模块化的 RAG SDK。关键架构问题如下：

1. Monorepo 还是多仓库？
2. 包边界 — 什么内容放在哪里？
3. 包之间如何通信？
4. 使用什么构建系统和工具？

## 决策

### 1. 使用 pnpm Workspaces 的 Monorepo

**选择：** pnpm workspace monorepo

**理由：**
- 所有包统一版本管理
- 简化跨包开发（无需 npm link）
- 共享工具配置
- 跨包的原子提交

**被否决的替代方案：**
- **多仓库** — 对于学习项目来说开销太大；跨包变更变得麻烦
- **npm workspaces** — pnpm 提供更好的依赖隔离和磁盘效率
- **yarn workspaces** — pnpm 的严格依赖解析更适合 SDK 开发

### 2. 七个包的划分

按领域职责拆分包：

| 包 | 职责 |
|---------|---------------|
| `core` | 共享类型、接口、抽象 |
| `adapters` | 提供商集成 |
| `indexing` | 文档处理 |
| `runtime` | 流水线编排 |
| `eval` | 评估 |
| `observability` | 监控 |
| `utils` | 共享工具函数 |

**理由：** 每个包代表一个独立的关注点，可以单独替换或使用。core 包强制接口边界，防止循环依赖。

### 3. 基于接口的通信

所有跨包通信通过 `@rag-sdk/core` 中定义的接口进行。

**理由：**
- 各包可以独立开发和测试
- 新的提供商实现不需要修改流水线逻辑
- 清晰的契约边界使并行开发成为可能

### 4. 仅支持 ESM 模块

所有包使用 `"type": "module"` 和 ESM import/export。

**理由：**
- 现代 Node.js（18+）完全支持 ESM
- 更好的 tree-shaking 以实现打包优化
- 与 TypeScript 生态方向一致
- 简化模块系统（无需同时支持 CJS/ESM）

### 5. TypeScript 严格模式

所有包的 tsconfig 使用 `strict: true`。

**理由：**
- 早期捕获类型错误
- 提供更好的 IDE 支持和开发体验
- 与 SDK 的"类型安全优先"理念一致

## 影响

### 正面影响

- 清晰的关注点分离
- 各包可以独立开发
- 容易添加新提供商（新适配器 = 新文件）
- 类型安全早期捕获集成问题

### 负面影响

- Monorepo 需要 pnpm；团队必须使用 pnpm
- 对于学习项目，七个包可能显得过度设计
- Core 中的接口变更需要在多个包中更新

### 缓解措施

- Core 接口应精心设计以减少变更
- 使用 `pnpm --filter` 进行有针对性的包操作
- 保持 core 轻量 — 仅包含跨包边界所需的内容
